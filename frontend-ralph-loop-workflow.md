# 🚀 融合式单/多代理前端开发 Ralph Loop 工作流

## 🎯 核心目标

结合**多代理职责分离**与**单代理多模态集成**的优势，构建一个可配置的、高度自治的前端开发工作流。该系统实现完整的 **Ralph Loop（持续迭代循环）**，具备：
- **多代理协同**：规划器、生成器、评估器的专业分工与对抗验证
- **多模态感知**：集成 Figma MCP（设计规范提取）和 Playwright MCP（视觉/功能验证）
- **防死循环机制**：熔断机制与上下文压缩，确保长时间稳定运行
- **灵活部署**：支持单代理模式（轻量级）和多代理模式（复杂项目）

---

## 🏛️ 架构模式选择

### 模式一：多代理协同模式（推荐用于复杂项目）
```
🧠 规划器 (Planner) → 💻 生成器 (Generator) ↔ ⚖️ 评估器 (Evaluator)
```
- **规划器**：项目初始化、需求拆解、Figma设计规范提取
- **生成器**：编码实现、即时上下文检索、基础自检
- **评估器**：Playwright自动化测试、视觉审美评审、功能验证

### 模式二：单代理全能模式（轻量级项目）
```
🤖 全能代理 (Unified Agent) → [设计溯源 → 编码 → 双重验证 → 熔断]
```
- 单个代理执行完整工作流
- 适用于功能简单、迭代快速的场景

## 🔧 核心工具与依赖

### MCP 集成层
1. **Figma MCP**：读取设计稿的图层结构、Auto Layout、颜色/字体 Token、间距规范
2. **Playwright MCP**：无头浏览器操作、DOM计算样式获取、页面截图、交互测试

### 本地工程层
- **Claude Code**：代理执行环境
- **Git**：版本控制与状态存档
- **Node.js/NPM**：前端开发环境
- **一键验证脚本**：`npm run verify`（包含 TypeScript 编译与 ESLint）

### 外部记忆系统
- `feature_list.json`：结构化任务清单与状态机
- `claude-progress.txt`：代理跨会话记忆与交接笔记
- `design_spec.json`：从 Figma 提取的设计规范缓存

## 🔄 完整工作流：多代理协同模式

### Phase 0：模式选择与环境准备
1. **人类输入**：提供项目描述、Figma链接、模式选择（单/多代理）
2. **权限预授权**：以 `claude --yes` 模式启动，避免交互阻塞
3. **MCP 连通性验证**：确保 Figma 和 Playwright MCP 可正常访问

### Phase 1：项目初始化（规划器主导）
*目标：建立标准化开发环境，提取设计规范，拆解任务。*

1. **需求分析与设计溯源**
   - 解析人类输入的 PRD
   - 调用 Figma MCP 提取设计规范，保存到 `design_spec.json`
   - 定义统一的视觉设计语言（Design Language）

2. **任务拆解与结构化**
   - 生成 `feature_list.json`，将项目拆解为原子级特性
   - 每个特性包含：`id`, `name`, `description`, `priority`, `passes`（初始为 `false`）, `design_references`

3. **环境搭建**
   - 生成 `init.sh`：一键启动本地开发服务器
   - 创建 `claude-progress.txt`：初始化记忆文件
   - 执行 `git init` 与 Initial Commit，封存初始状态

### Phase 2：冲刺交接与上下文重置
*目标：消除上下文焦虑，确保代理像轮班工人一样清晰交接。*

1. **开启新代理会话**：根据模式选择启动生成器（多代理）或全能代理（单代理）
2. **快速进入状态**：
   - 读取 `claude-progress.txt` 获取项目记忆
   - 查看最新 `git log` 了解代码状态
   - 扫描 `feature_list.json`，选择**优先级最高且 `passes: false`** 的特性
3. **签订"冲刺契约"**：
   - 生成器/全能代理提出技术实现方案和"完成标准（Done Criteria）"
   - 评估器（多代理模式）或自检逻辑（单代理模式）审查契约
   - 双方达成一致后进入开发

### Phase 3：核心 Ralph Loop（生成-评估对抗循环）

#### 步骤 3.1：设计指导编码
- 生成器读取 `design_spec.json` 中对应特性的设计规范
- 使用窄范围工具（`grep`, `cat`）动态定位需要修改的文件
- 编写代码，确保符合设计规范

#### 步骤 3.2：本地环境启动
- 运行 `init.sh` 启动本地开发服务器
- 验证服务正常启动，无编译错误

#### 步骤 3.3：双重客观验证
**A. 代码静态验证**
- 执行 `npm run verify`（TypeScript + ESLint）
- 如有错误，自动修复并记录到重试计数器

**B. 多模态视觉验证**
- 调用 Playwright MCP 访问本地服务（如 `http://localhost:5173`）
- **四维严格打分**：
  1. **设计一致性**：比对 DOM 计算样式与 Figma 规范（颜色、字体、间距）
  2. **视觉审美**：截图审查布局对齐、颜色对比度、排版层级
  3. **交互功能**：按钮点击、表单输入、路由跳转等交互测试
  4. **响应式适配**：多视口尺寸测试（桌面、平板、移动端）

#### 步骤 3.4：决策分叉与熔断机制

**✅ 通过 (Pass)**
- 跳出当前特性循环，进入 Phase 4

**🔄 未通过 (Fail) - 自动修复尝试**
- 评估器输出详细批评报告（含截图和样式差异数据）
- 生成器根据反馈决定修复策略：
  - **局部微调 (Refine)**：小范围样式/逻辑调整
  - **美学重构 (Pivot)**：彻底重新设计实现
- **熔断保护**：设置 `MAX_RETRIES = 3`
  - 每次重试在回复首行声明：`[自动修复尝试: X/3]`
  - 达到3次阈值，触发熔断机制

**🚨 熔断触发**
当满足以下任一条件时触发熔断：
1. 连续3次修复尝试后仍失败
2. 检测到死循环模式（反复修改同一处逻辑）
3. 发现无法自动解决的底层问题（依赖冲突、配置问题）

**熔断后行动**：
1. 立即停止所有修改
2. 生成《异常诊断与接管报告》：
   ```
   🚨 任务陷入死循环，已触发熔断机制。
   
   [根因分析]
   1. 精确指出根本原因
   2. 描述已尝试的修复方案及失败原因
   
   [给人类的接管建议]
   1. 需要人工检查的具体文件/行号
   2. 明确的修复建议
   ```
3. 保持当前工程状态，等待人类接管

### Phase 4：状态存档与上下文压缩
*目标：安全封存成果，提炼核心记忆，为下一个特性开发铺平道路。*

1. **更新全局状态**：
   - 将 `feature_list.json` 中当前特性的 `passes` 更新为 `true`
   - 记录通过时间、最终实现方案摘要

2. **结构化记忆更新**：
   - 在 `claude-progress.txt` 中写入高度浓缩的总结
   - 格式：`[日期] 特性ID：关键实现点、注意事项、后续建议`

3. **Git 里程碑提交**：
   - 执行描述清晰的 Git Commit
   - 提交消息格式：`feat(scope): 特性名称 - 关键变更摘要`

4. **上下文压缩**：
   - 清理当前会话的冗长日志和中间文件
   - 仅携带更新后的记忆文件进入下一个循环

5. **循环检测**：
   - 检查 `feature_list.json` 是否仍有 `passes: false` 的特性
   - 如有，返回 **Phase 2** 继续下一个特性
   - 如无，进入项目收尾阶段

### Phase 5：项目收尾与交付
1. **端到端集成测试**：运行完整的 Playwright E2E 测试套件
2. **性能与可访问性检查**：Lighthouse 审计、键盘导航测试
3. **文档生成**：自动生成组件文档、样式指南
4. **最终提交与标签**：Git Tag 标记版本，准备部署

## 🔄 单代理模式简化流程

对于单代理模式，工作流简化为以下循环：

```
设计溯源 → 编码实现 → 双重验证 → [通过/修复] → 状态存档
```

**关键调整**：
1. 代理内部维护 `feature_list.json` 和记忆文件
2. 自检逻辑替代评估器的对抗验证
3. 熔断机制同样适用，防止死循环
4. 每个特性完成后进行 Git 提交和记忆更新

## 🛡️ 质量控制与容错机制

### 防崩溃策略
1. **Git 安全网**：每个特性完成后立即提交，可随时回退到上一个稳定状态
2. **注意力预算管理**：强制使用窄范围文件操作，避免将整个项目塞入上下文
3. **记忆外部化**：所有长期记忆存储在文件中，不依赖会话上下文

### 熔断升级策略
1. **一级熔断**：单特性重试超限（3次）
2. **二级熔断**：连续3个特性触发一级熔断
3. **三级熔断**：检测到系统性架构问题，建议项目重构

### 人类接管接口
熔断后提供清晰的接管路径：
1. **问题定位**：精确到文件、行号、CSS选择器
2. **环境状态**：当前运行的服务、安装的依赖、Git状态
3. **建议方案**：具体的代码修改建议或配置调整

## 🚀 快速启动指南

### 前置条件
1. 安装 Claude Code 并配置 MCP 服务器
2. 准备前端项目模板（React/Vue等）
3. 获取 Figma Personal Access Token
4. 安装 Playwright 测试环境

### 启动命令
```bash
# 多代理模式启动
claude --yes --system-prompt "./.claude-multi-agent.md"

# 单代理模式启动  
claude --yes --system-prompt "./.claude-unified-agent.md"
```

### 系统指令模板
创建 `.claude-multi-agent.md` 文件，包含相应模式的完整工作流指令。
