### Phase 3: Local Environment Verification

#### Step 3.1: Start/Restart Development Server
1. If the service is not running: execute `./init.sh`
2. If the service is already running: ensure code changes have hot‑reloaded
3. Verify the console has no errors

#### Step 3.2: Dual‑Objective Verification Loop

**Set circuit‑breaker counter**: `MAX_RETRIES = 3`, current attempt: `[X/3]`

##### A. Code Static Verification (Must execute after each modification)
1. Run `npm run verify` (includes `tsc --noEmit` and `eslint`)
2. **Result handling**:
   - ✅ **Pass**: Proceed to visual verification
   - ❌ **Fail**: Automatically fix errors, counter+1, record in memory file

##### B. Multimodal Visual Verification (Execute after code verification passes)
1. **Call Playwright MCP** to access the local service (e.g., `http://localhost:5173`)
2. **Four‑dimensional strict testing**:

   **a) Design‑Consistency Test**
   - Obtain `getComputedStyle()` of key DOM nodes
   - Compare color, font, spacing, dimensions with Figma specifications
   - Allow ±5% visual variance (screen‑rendering vs. design‑tool differences)

   **b) Visual‑Aesthetic Review**
   - Screenshot review of overall layout, alignment, hierarchy
   - Check color contrast (WCAG AA standard)
   - Evaluate visual appeal, avoid “AI‑looking” design

   **c) Interactive‑Functionality Test**
   - Actually click buttons, links, form controls
   - Test form validation, state toggles, animation effects
   - Verify route navigation, data‑loading states

   **d) Responsive‑Adaptation Test**
   - Desktop (≥1024px), tablet (768px), mobile (≤480px)
   - Check layout changes, font‑size adjustments, interaction adaptation

3. **Verification‑result handling**:

   **✅ Fully Pass**
   - All test items meet acceptance criteria
   - Exit verification loop, proceed to Phase 4

   **🔄 Partial Failure**
   - Generate a detailed discrepancy report:
     ```
     [Verification Failed] FeatureID: feature‑001
     Failed items:
     1. Button background color: #FF3300 (actual) ≠ #FF0000 (design)
     2. Mobile navigation menu does not collapse after click
     3. Font line‑height: 1.2 (actual) ≠ 1.5 (design)
     Suggested fixes: Adjust CSS variables, fix event listener
     ```
   - Counter+1, enter repair phase

#### Step 3.3: Intelligent Repair Strategy

Choose a repair strategy based on failure type:

1. **Local Refinement (Refine)** – Suitable for small‑scope style/logic issues
   - Adjust CSS property values
   - Fix event‑handler functions
   - Fine‑tune component props

2. **Aesthetic Pivot (Pivot)** – Suitable for overall design mismatch
   - Re‑evaluate design‑implementation approach
   - Consult Figma specifications, confirm understanding is correct
   - Consider alternative layout/interaction solutions

3. **Architectural Adjustment** – Suitable for technical debt or dependency issues
   - Refactor component structure
   - Extract shared logic
   - Update dependency versions

4. **Use superpowers skill to assist repairs** – Suitable for complex problems or need for intelligent optimization
   - Call `Skill` tool to execute `superpowers` skill
   - Intelligently fix code issues based on verification feedback
   - Automatically optimize code structure and implementation

**After each repair**:
1. Re‑run **Code Static Verification**
2. If passed, re‑run **Multimodal Visual Verification**
3. Update attempt counter: `[Auto‑repair attempt: X/3]`

### Phase 4: Circuit‑Breaker Protection Mechanism

#### 🚨 Circuit‑Breaker Trigger Conditions (Trigger if any is satisfied)
1. `[Auto‑repair attempt: 3/3]` still fails verification
2. Dead‑loop pattern detected (modifying the same CSS property 3 times in a row)
3. Discovery of systemic problems that cannot be resolved automatically:
   - Dependency version conflicts
   - Build‑configuration errors
   - Cross‑origin or network issues
   - Design‑specification contradictions

#### Post‑Circuit‑Breaker Standard Operating Procedure
1. **Immediately stop all modifications**, maintain the current engineering state
2. **Generate “Abnormal Diagnosis & Handover Report”**:

```markdown
🚨 Task stuck in dead loop, circuit‑breaker mechanism triggered.

[Root‑Cause Analysis]
1. (Precise technical reason, e.g., third‑party library’s global‑style override causing z‑index failure)
2. (Attempted repair solutions: tried modifying z‑index, position, isolation properties, all ineffective)
3. (Why it cannot be resolved automatically: requires modifying third‑party‑library configuration or global‑style reset)

[Current Engineering State]
- Local service: http://localhost:5173 (running/stopped)
- Git status: (current branch, uncommitted changes)
- Latest error: (console‑error log)

[Handover Suggestions for Human]
1. Please check file: `src/components/ChartContainer.tsx:45‑60`
2. Recommended action: Add `zIndex: 0` configuration to third‑party chart‑component options
3. Or modify: `src/styles/globals.css` add `* { isolation: isolate; }`
4. After takeover, run: `npm run verify && re‑test`
```

3. **Wait for human takeover**, do not perform any further modifications
4. **Keep the environment running** to facilitate human debugging

## 🛡️ Core Behavioral Guidelines

### 1. Attention‑Budget Management
- **Prohibited**: Loading the entire project directory at once
- **Required**: Use `grep`, `cat`, `find` for precise location
- **Principle**: Keep working context lightweight, rely on external memory files

### 2. Git Safety‑Net Practices
- **Must** commit after each feature completion
- Commit messages should clearly describe **why** the change was made, not just what was done
- When encountering uncontrollable errors, `git reset --hard HEAD` to revert to the previous state

### 3. Circuit‑Breaker Mechanism Priority
- Strictly count retry attempts; do not continue after `[3/3]`
- Circuit‑breaker reports must be specific and actionable
- Maintain environment state to facilitate human takeover

### 4. Design‑Specification Compliance
- Prefer using design tokens from `design_spec.json`
- If design‑specification contradictions are found, record and request clarification
- For aesthetic judgments, treat the Figma design file as authoritative

### 5. Progressive Complexity Management
- Implement core functionality first, then add enhancements
- Keep component interfaces simple, expand gradually
- Avoid premature optimization; prioritize correctness
